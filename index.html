Unified Support Experience

Unified support experience for both agents and clients

Vision — Customer

One continuous conversation, from first contact to resolution

Today, customers navigate disconnected channels when reaching out to support. Our goal: a unified experience where every interaction — AI, chat, email, phone — lives in a single, continuous conversation thread.

Today

🔌
Fragmented journey Customers move from AI to live-chat to email tickets to phone calls — each in a separate interface with no continuity.
🔄
Lost context Switching channels can force customers to re-explain their issue, as agents may not have visibility into previous interactions.
⏳
Premature closure In some cases, live-chat sessions end before the issue is fully resolved. A separate ticket is then created to continue — breaking the conversation flow.
🔍
No unified tracking Customers have no single place to view and follow up on all their support requests.
→

Tomorrow

💬
One conversation thread Every interaction — AI, chat, email, phone — lives in a single continuous thread. Like messaging a friend.
🔄
Seamless sync-to-async transition Customers can leave and come back anytime. The conversation stays open — no timeout, no lost progress.
📋
Full context preserved Agents see the complete history regardless of channel. No need to repeat information.
📱
Everything in the PayFit app Customers track all their support requests end-to-end, directly within the application.
Vision — Agent

A unified workspace, built for resolution

Agents spend less time navigating between tools and more time solving problems. One workspace, full context, AI-powered assistance.

Today

🗂
Scattered tooling Agents juggle between Salesforce Cases, Salesforce Live Chat, and the PayFit app. Each channel has its own interface.
🛑
Real-time pressure Live-chat demands immediate responses, even when the issue requires deeper investigation or collaboration.
🔍
Fragmented context Copilot history lives in one tab, the case in another. Agents manually piece together the full picture.
📋
Routing management Routing is limited, making it harder to consistently assign each request to the best-qualified agent for the customer's specific need.
→

Tomorrow

💻
Work where the context lives Agents operate directly inside the PayFit app, next to the customer's data. No more switching to Salesforce to respond.
⏱
Respond on your own terms Reply in real-time or take the time to investigate. The hybrid model removes the pressure of forced synchronous chat.
🤖
AI-powered assistance PayFit Copilot provides contextual suggestions and surfaces the complete conversation history to accelerate resolution.
📥
Intelligent routing Support requests are automatically pushed to the right agent via Amazon Connect. No more manual queue browsing.

Migration strategy

A progressive migration, not a big bang

We are intentionally moving in three phases so we can deliver value early, reduce delivery risk, and adapt effort over time without destabilizing teams.

What we avoid

🛑
No all-at-once cutover We do not switch every interaction type and workflow in one release.
⏱
No delayed outcomes We do not wait for the full end-state to start delivering customer and agent improvements.
→

How we execute

1
Phase 1: deliver immediate value Remove inbox dependency and bring case visibility into Support Request in the app.
2
Phase 2: unify the user journey Merge sync and async interactions into one Support Request with continuity of context.
3
Phase 3: complete routing migration Finalize CCASS migration and move agents to the new console when the foundations are proven.

Alternative path (not recommended)

No Support Request: direct Salesforce Cases + Amazon Connect Live-chat

This alternative goes against the target vision. Without an intermediate Support Request as source of truth, we cannot migrate async cases to Amazon Connect and keep Salesforce only for the required agent populations. Amazon Connect can only cover live-chat and callbacks, and we would need to fetch multiple data sources to recreate partial continuity for customers.

Why we do not recommend this alternative

  • No intermediate entity — no Support Request means no stable source of truth for both Salesforce and Connect
  • Async cases cannot migrate to Amazon Connect — without Support Request as intermediate entity, case handling must remain on Salesforce
  • Dual-write architecture — the console must write every update to two case systems in parallel
  • Higher mismatch risk — if one write fails, cases diverge and require reconciliation
  • Higher maintenance cost — permanent sync, mapping, and recovery logic becomes core platform burden
  • Weaker customer continuity — we must query multiple data sources to rebuild journey context
  • Strategic dead-end — limits evolution toward Agent Cockpit and unified sync/async journey
